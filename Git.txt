🌿 Why people create different branches

A branch is like a separate workspace where you can work without affecting the main code.

🎯 Main reasons
1️⃣ Work on features safely

Instead of breaking your main code:

main (stable)
   ↓
feature/login-page

You can:

experiment
make mistakes
test freely

👉 without touching main

2️⃣ Avoid breaking production code
main (or master) = stable, working code
Other branches = work in progress
3️⃣ Work in teams

Different people work on different branches:

main
├── feature/images
├── feature/navbar
└── bugfix/login-error

👉 No conflicts while developing

4️⃣ Organize work

Branches are often named like:

feature/add-images
bugfix/fix-header
hotfix/security-patch
🔀 How to merge branches

Let’s say you worked on:

feature/images

and now want to merge into main.

✅ Step 1 — Switch to main
git checkout main
✅ Step 2 — Merge the branch
git merge feature/images

👉 This brings all changes into main.

✅ Step 3 — Push to GitHub
git push origin main
🔍 What happens during merge

Git combines histories:

Before:
main:     A---B
feature:       \---C---D

After merge:
main:     A---B-------E
                   /  
feature:       C---D
⚠️ Merge conflicts (important)

Sometimes Git can’t merge automatically:

CONFLICT in app.py

You must:

Open the file
Fix the conflict manually
Then:
git add .
git commit
🔁 Alternative: Pull Request (GitHub way)

Instead of merging locally:

Push your branch:

git push origin feature/images
Go to GitHub
Click “Compare & Pull Request”
Merge there

👉 This is the most common team workflow.

🧠 Simple analogy
main → official version of your project
branches → parallel universes where you try things
🚀 Typical workflow
git checkout -b feature/images   # create branch
# make changes
git add .
git commit -m "Add images"
git push origin feature/images

# later merge
git checkout main
git merge feature/images
git push origin main
💡 Pro tip

After merging, you can delete the branch:

git branch -d feature/images
